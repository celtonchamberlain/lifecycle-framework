# Lifecycle Framework — Canonical Definitions

> The authoritative definitions behind the plugin: **(1)** the documents the scaffolder creates, **(2)** the
> lifecycle, **(3)** the roles. This is the source of truth the `data/templates/` and agent files are generated from.
> Reflects the framework's defaults: models all-Opus, naming role-descriptive, separation-of-duties collapsed +
> freeze enforced, tracker Jira/Linear asked at init.

---

# PART 1 — Scaffolded documents and their definitions

Grouped by function. Each entry: **what it is · why it exists (the failure it prevents) · who writes it · when ·
protection**. "Protected" = the PM never edits it directly; it proposes a change during close and the human
authorizes. "Append-only" = entries are added, never edited or deleted.

## A. Persistent governance docs (outlive every task)

These three hold the project's architecture, its history, and its intent. They survive across all tasks.

### `docs/strategy.md` — **PROTECTED**
- **What:** The goal and vision. Product + technical strategy, the decision log (the "why we do X this way"), and a
  risk register.
- **Why:** A single source of truth for intent and rationale. Without it, every spec re-litigates decisions already
  made and the project drifts from its purpose.
- **Writes:** Human + PM (proposal-mode). **When:** at init, then on decisions of record. **Protection:** protected.

### `docs/architecture.md` — **PROTECTED**
- **What:** How the system is built **today** — ERD, application architecture, data flow, tech stack, the live
  topology. Dynamic: it changes as the system does.
- **Why:** Prevents writing architectural assumptions from memory. It is **re-audited from live sources** (not
  recalled) on a trigger matrix at close, so it never goes stale silently.
- **Writes:** PM (proposal-mode), re-audited at close. **When:** at init, then when the system changes.
  **Protection:** protected.

### `docs/project_chronicle.md` — **APPEND-ONLY**
- **What:** A running record of the important decisions and findings across the whole project. One entry per
  completed task: what was done, key findings, decisions, questions raised/resolved, next action.
- **Why:** A ticket spans many tasks; memory does not. The chronicle keeps the reasoning. It is **cited as incident
  history** by the code-reviewer and devils-advocate to ground their objections in what actually happened before.
- **Writes:** PM (via close-task) / corpus-steward. **When:** at every close, append only. **Protection:** append-only.

## B. Recoverable state files (state is files, not memory)

The project's current state must be reconstructable from a handful of files, so any session can resume cold.

### `TODO.md` (repo root) — **PROTECTED**
- **What:** The dependency-ordered task route, mirrored from the tracker.
- **Why:** The single ordered "what's next", independent of any chat context. Mirrors the tracker so the plan
  survives outside it.
- **Writes:** PM, status-sync only via close-task. **Protection:** protected.

### `docs/context_snapshot.md`
- **What:** The single "where we are" handoff for the next session. Current state only (not history), kept short,
  facts only. Overwritten each close.
- **Why:** Lets the next session resume **without re-discovery**. The deliberate overwrite keeps it a snapshot, not a log.
- **Writes:** close-task (overwrite). **When:** every session boundary. Usually gitignored.

### `docs/activity_log.jsonl` — **APPEND-ONLY**
- **What:** The machine-readable audit trail. One JSON line per lifecycle step plus one per commit. Fields:
  `ts, session, ticket, branch, pr_url, step, event, agent, model, summary, artifacts, tracker_comment_id`.
- **Why:** The granular, queryable record of *what happened, by whom, when, with which model*. The "every step
  leaves a trace" guarantee. The `tracker_comment_id` cross-references the human-readable tracker comment
  (the other half of the dual-write).
- **Writes:** `/log-activity` + the post-commit git hook. **When:** after every step / commit. **Protection:** append-only.

### `.claude/log_session`
- **What:** A tiny JSON state file `{ id, branch, issue }` holding the current session id, branch, and active ticket.
- **Why:** Gives the log skills a stable session id to stamp every JSONL line with, so a session's steps are cleanly
  segmented in the log.
- **Writes:** session start / log skills.

## C. Queryable corpus index

### `docs/INDEX.md` + `docs/index.json` — autogenerated
- **What:** A generated index of every governed `.md`: its type, status, authority, and relationships
  (`depends_on` / `affects`), built by `build-index.mjs`.
- **Why:** Turns loose files into a machine-checkable knowledge base. The real value is **seeing relationships** —
  what depends on what you're about to change — not just finding files. Freshness is gated by the pre-push hook and CI
  (the push is blocked if the INDEX drifts from source).
- **Writes:** `build-index.mjs` (never by hand). **When:** pre-push, CI, and `/audit-corpus`.

## D. Lifecycle artifacts (`docs/claude_tasks/` and siblings)

One place per artifact type, so the folder is a corpus, not a pile.

### `docs/claude_tasks/<NN_slug>.md` — the **spec**
- **What:** One spec per ticket: scope, out-of-scope, approach, **acceptance criteria**, known risks.
- **Why:** The contract the work is built and judged against. The expanded form of the rough brief.
- **Writes:** PM (Step 3). **Frozen at Step 6** before implementation.

### `docs/claude_tasks/alpha_tests/` — the **frozen acceptance tests**
- **What:** The pre-committed, pass/fail acceptance test, designed from the spec **before any code exists**, then
  **sealed (frozen)**.
- **Why:** Test-first applied to review. Because the test is frozen before the code, the runner **cannot tailor it to
  the result** → the verdict is binary, bias-free. The FREEZE RULE: a test that can change after the code is seen is
  not bias-free. (Enforced: `/audit-corpus` flags a sealed alpha test modified after sealing as a High finding.)
- **Writes:** DA (Step 4 design, Step 6 seal). **The DE never touches this directory** (the "alpha-test wall").

### `docs/claude_tasks/reviews/` — spec & code reviews
- **What:** The written reviews: `review_de` (feasibility), `review_devils-advocate` (ranked objections),
  `review_code-reviewer` (deploy verdict), `review_da` (data validation).
- **Why:** Review output is an artifact, not a chat message — auditable and frontmatter-linked to its spec.
- **Writes:** the reviewing role. **When:** Steps 5 and 8.

### `docs/claude_tasks/reports/` — task reports
- **What:** The DE's implementation report: what was built, decisions, deviations, self-review checklist results.
- **Why:** The record the PM reviews against acceptance criteria at Step 8. **Writes:** DE (Step 7).

### `docs/claude_tasks/council/` — council decision docs
- **What:** One document per convened Council: the question, the positions, the devils-advocate attack, the decision,
  and its **reopen criteria**.
- **Why:** Contested/architectural/irreversible decisions get a durable, reviewable record — including under what
  conditions the decision may be reopened. **Writes:** `/council`.

### `docs/audits/`
- **What:** Output home for `/audit-corpus` health reports (tiered High/Medium/Low) and the corpus-steward's
  archival manifests. **Why:** keeps corpus-health output as dated, reviewable artifacts.

### `docs/knowledge_base/`
- **What:** Curated domain knowledge the agents read: glossary / metric definitions, data dictionary, business
  rules, chart guidance. **Seeded at init** from any initial docs the user provides.
- **Why:** Grounds the agents in domain truth so specs and analysis are built against reality, not assumptions.
  **Writes:** human-curated (seeded at init), maintained over time.

### `docs/_templates/`
- **What:** The frontmatter-valid templates `/new-doc` scaffolds from: `spec, alpha_test, report, review_de,
  review_da, review_code-reviewer, review_devils-advocate, council, handoff`.
- **Why:** Every governed doc starts valid (correct frontmatter, correct shape) → the corpus stays machine-checkable.
  Scaffolded per-project so projects can extend them.

## E. Configuration & operational files

### `CLAUDE.md` (repo root)
- **What:** The per-project operating contract read at every session start: communication principle, model policy,
  session-start protocol, lifecycle numbering, role roster, stack standards, tracker identity, build commands.
- **Why:** The single entry point that tells every agent how *this* project works. (A plugin-root CLAUDE.md is not
  loaded as context — it must live in the project, which is why it's scaffolded.)

### `.claude/settings.json` (committed)
- **What:** Project config: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, tracker type + connection, stack, enabled MCP
  servers, the scoped permission allowlist, and the PreToolUse hook wiring.
- **Why:** **Committed** (not `.local`) so spawned teammates and CI inherit the Agent Teams flag and permissions —
  otherwise teammates block on prompts mid-lifecycle or the orchestration silently collapses to a single agent.

### `.claude/.local/secrets.env` (gitignored)
- **What:** A stub for project-local secrets: tracker API token, GitHub token, per-project memory path, MCP creds.
- **Why:** Keeps credentials out of git. Never committed. (Never hardcode credentials.)

### `.claude/rules/*.md`
- **What:** The governance rules read at session start. Definitions:
  - `code-style.md` — file organization, naming, comment intent (WHY not WHAT), error handling.
  - `dependency-governance.md` — gate before adding any dependency: already-present check, vanilla-first,
    verify/justify/pin, escalate non-trivial via `/council`.
  - `frontmatter.md` — **the frontmatter taxonomy SSOT**: field schema, `type`/`status`/`authority` enums,
    `depends_on`/`affects`, validation rules. Drives the validator, husky, and CI. Required for the governed corpus.
  - `git-workflow.md` — conventional commits, `1 ticket = 1 branch = 1 PR`, never force-push.
  - `mcp-usage.md` — when/how to use each MCP server; Memory protocol at session start + after every task.
  - `search-first.md` — read a file and its importers/dependents/tests **before** editing it.
  - `security.md` — no hardcoded secrets, parameterized queries, input validation, no leaked internal errors.
  - `testing.md` — test priority order, organization, independence + determinism.

### `.husky/` git hooks (project-local)
- **What:** `pre-commit` (frontmatter validate `--warn-only`), `post-commit` (auto-append a `committed` JSONL line),
  `pre-push` (rebuild INDEX, block on drift).
- **Why:** Enforces traceability + corpus freshness at the git boundary, automatically.

### `.github/workflows/lint-frontmatter.yml` (+ optional CI by stack)
- **What:** MANDATORY CI: strict frontmatter validation (no `--warn-only`) + INDEX-freshness verify. Optional add-ons:
  `claude-code-review` (always-on bias-free PR-diff review), schema/catalog snapshot, dead-code, lint gates.
- **Why:** The third, always-on, bias-free review layer (after DE self-review and PM review) on the PR diff.

### `scripts/*.mjs` + `corpus.config.mjs`
- **What:** The corpus engine: `validate-frontmatter`, `build-index`, `log-commit-activity`, `affects-lookup`,
  `_corpus-walk`, plus per-project constants in `corpus.config.mjs`.
- **Why:** The tooling the skills, hooks, and CI call. **Scaffolded into the repo** (not plugin-resident) because
  husky/CI invoke them by relative path and `${CLAUDE_PLUGIN_ROOT}` doesn't resolve there.

# PART 2 — The lifecycle and its definitions

**The rule:** one Claude Code session runs **exactly one task**, Brief → Close — never a whole ticket. A ticket is
the umbrella (many tasks → many sessions). A task too big for one session uses a *transitory close* at each boundary
that persists the snapshot + memory, so the next session resumes exactly where this one stopped.

Numbering is the canonical 1–10 (the deck). `/log-activity` fires after **every** step; `/council` may be convened at
**any** point. Three phases: **Frame (1–4) · Challenge then build (5–7) · Verify and close (8–10)**.

| # | Step | Actor | Definition |
|---|------|-------|------------|
| 1 | **Brief** | Human | A short, intentionally rough ticket: the goal and the motivation. Detail is the team's job, not the brief's. |
| 2 | **Data analysis** | DA | The DA profiles the **real data** before any spec is written, so the spec is grounded in reality, not assumptions. Exact numbers, full queries; read-only. |
| 3 | **Spec expanded** | PM | Brief + analysis become a full spec: scope, out-of-scope, approach, **acceptance criteria**, known risks. Written to `claude_tasks/`. |
| 4 | **Alpha test** | DA | The DA designs the pre-committed **acceptance test against the spec, before any implementation exists**. Test-first. |
| 5 | **Spec reviewed** | DE + devils-advocate | The DE checks feasibility/correctness with executed read-only probes. **In parallel**, the devils-advocate challenges the spec with ranked, evidence-grounded objections. No spec reaches implementation unchallenged. |
| 6 | **Resolve** | PM | The PM addresses every objection. If the spec changes, the alpha test is refreshed. The spec is then **FROZEN** — this is the freeze point, before implementation. |
| 7 | **Implementation** | DE | The DE builds against the frozen spec, runs a self-review checklist (signatures, no hardcoded config, security, error handling, tests pass), writes the task report. **The code-reviewer gate runs here** before any deploy/commit (any CRITICAL → NOT SAFE). |
| 8 | **Review** | PM + DA | The PM checks the report against acceptance criteria. The **DA runs the frozen alpha test** — binary pass/fail (collapsed SoD; in strict mode a separate `data-reviewer` runs it). |
| 9 | **Validation** | Human | The human does the final acceptance check on the live result, and either accepts it or sends it back. *(The one human gate — see open decision.)* |
| 10 | **Close** | PM | `/close-task` writes the final log entry, refreshes the snapshot + TODO, appends to the chronicle, conditionally re-audits architecture, writes to MCP memory, and moves the ticket to Done. |

**Bias control (the spine):** the test is written before the code (Step 4), frozen before implementation (Step 6),
and run against a result it could not influence (Step 8). **Separation of duties:** DA designs *and* (in collapsed
mode) runs the frozen test; the **DE never touches the test**; the code-reviewer reviews code, never fixes it.

**Three review layers:** ① DE self-review (in session, own checklist) → ② PM review (in session, vs spec) → ③ CI
code-review (on the PR diff, always-on, bias-free).

**Traceability (dual-write):** after each step, `/log-activity` appends the JSONL line and `/log-tracker` (at
milestones) posts the mirrored tracker comment, cross-referenced by id.

# PART 3 — Role definitions

The team is **role-specialized, not one generalist**. Each agent has an explicit *can* and *cannot*, a fitting
model (cost follows criticality), and is spawned by the PM as an Agent-Teams teammate in its own context window.

## Core five (always shipped)

### PM — `pm` · model: session (Opus)
- **Mandate:** Lead and orchestrator; the default role of every session. Owns the full lifecycle for one task:
  expands the brief into a spec, spawns DE/DA/code-reviewer/devils-advocate as teammates, parallelizes independent
  sub-tasks, resolves review findings, runs the gate, triggers human validation, runs `/close-task`. Enforces
  logging after every step and plain-language communication.
- **Can:** orchestrate, spec, resolve, decide (with council when contested), propose protected-doc changes.
- **Cannot:** write implementation code; design or run the alpha test; edit protected docs without authorization.

### DE — `de` · model: Opus
- **Mandate:** Development Engineer. Two lifecycle jobs: **Step 5** spec feasibility review (with executed read-only
  probes; pushback-with-evidence is a duty) and **Step 7** implementation against the frozen spec + mandatory
  self-review checklist + task report.
- **Can:** review specs, implement, self-review, report. **Cannot:** write specs, make architectural decisions,
  modify `TODO.md`/`CLAUDE.md`, deploy, or **touch the alpha-test directory** (the "alpha-test wall").

### DA — `da` · model: Opus
- **Mandate:** Data / QA Analyst. Three lifecycle moments: **Step 2** data profiling before any spec, **Step 4**
  alpha-test design from the spec (+ Step 6 seal), **Step 8** binary frozen-test execution. Read-only on production
  data (SELECT only); reproducibility non-negotiable (exact numbers, full queries).
- **Can:** profile data, design + run the frozen test, validate outputs. **Cannot:** write app code, mutate data,
  make architectural decisions.

### code-reviewer — `code-reviewer` (alias: heimdall) · model: Opus
- **Mandate:** The code-review QA gate, mandatory before any deploy/commit of production code (docs-only changes
  exempt). **Read-only** (writes only its own review file). Static validation + a severity-tiered bug checklist
  (CRITICAL / WARNING / SUGGESTION / SCHEMA) encoding incident history → a binary verdict
  (SAFE / NON-BLOCKING / NOT SAFE). **Any CRITICAL = NOT SAFE TO DEPLOY.**
- **Can:** review code, emit a verdict, describe fixes as text. **Cannot:** fix bugs, write code, approve its own work.

### devils-advocate — `devils-advocate` (alias: loki) · model: Opus
- **Mandate:** Adversary. Stress-tests **reasoning, not code**. Fires automatically at Step 5 in parallel with the DE
  review, is a **mandatory participant on every Council**, and is optional before irreversible actions. Output: a
  ranked list of concrete objections (fragile assumptions, unhandled cases, failure modes, cheaper alternatives).
  **Grounded-Dissent Protocol:** every objection must cite evidence (file:line, data row, memory incident,
  architecture principle, hard rule, chronicle problem) or it is discarded.
- **Can:** attack, rank objections. **Cannot:** propose solutions, implement, or **ever approve**.

## Optional roles

### corpus-steward — `corpus-steward` (alias: mimir) · default: ON · model: Sonnet
- **Mandate:** Read-only docs-governance archival steward (**PROPOSAL MODE**). On demand, sweeps `docs/` and writes a
  dated archival/cleanup manifest (closed-ticket bundles, superseded docs, consolidation candidates), each with an
  exhaustive inbound-reference safety check and a never-archive exclusion set (protected files, SSOT with live
  consumers, open-ticket dependents). **Never moves or deletes** — the PM executes. Realizes the deck's deferred
  "Auditor" idea safely.

### data-reviewer — `data-reviewer` · default: OFF (strict-SoD opt-in) · model: Opus
- **Mandate:** A distinct QA agent that **runs the frozen alpha test at Step 8** while the DA only **designs** it
  (full separation of duties, per the deck). Enabled only when `separation_of_duties_mode = strict`.

### scout — `scout` · default: OFF · model: Sonnet
- **Mandate:** Pre-edit blast-radius / dependency analysis. Before an edit, maps inbound deps (who imports me =
  blast radius) + outbound deps, flags circular-dependency risk and boundary violations. Overlaps `affects-lookup.mjs`.

### dead-code-cleanup — `dead-code-cleanup` (alias: fenrir) · default: OFF · model: Sonnet
- **Mandate:** Dead-code hunter: unused functions/imports/modules, orphaned files, dead endpoints, stale
  TODO/FIXME. Read-only by default; deletes only on explicit approval, then re-validates. Overlaps the dead-code CI.

## Hard prerequisite

All orchestration depends on **Agent Teams** (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`). Without it the PM cannot
spawn teammates: the team collapses into a single agent and the lifecycle's separation of duties cannot hold.
